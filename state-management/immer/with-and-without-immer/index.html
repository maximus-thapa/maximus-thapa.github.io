    With and Without Immer

    Using Immer in reducers simplifies the handling of immutable state by allowing you to write "mutable" code that actually produces an immutable update. Here’s a comparison of using Immer versus not using it in Redux reducers:

    When you manage state immutably without Immer, you typically have to create new copies of the state for any updates. This often involves using methods like Object.assign, the spread operator, or libraries like immutable.js. Here’s an example:

    Example Reducer without Immer

    const initialState = {
    loading: false,
    data: null,
    error: null,
};

const dataReducer = (state = initialState, action) => {
    switch (action.type) {
        case FETCH_DATA_REQUEST:
            return {
                ...state,
                loading: true,
                error: null,
            };
        case FETCH_DATA_SUCCESS:
            return {
                ...state,
                loading: false,
                data: action.payload,
            };
        case FETCH_DATA_FAILURE:
            return {
                ...state,
                loading: false,
                error: action.payload,
            };
        default:
            return state;
    }
};

    Using Immer allows you to write code as if you're directly mutating the state, but it actually creates a new immutable state under the hood. This leads to cleaner, more readable code. Here’s the same example using Immer:

    Example Reducer with Immer

    import produce from 'immer';

const initialState = {
    loading: false,
    data: null,
    error: null,
};

const dataReducer = (state = initialState, action) =>
    produce(state, (draft) => {
        switch (action.type) {
            case FETCH_DATA_REQUEST:
                draft.loading = true;
                draft.error = null;
                break;
            case FETCH_DATA_SUCCESS:
                draft.loading = false;
                draft.data = action.payload;
                break;
            case FETCH_DATA_FAILURE:
                draft.loading = false;
                draft.error = action.payload;
                break;
            default:
                break;
        }
    });

    1. Readability:

      • Without Immer: You need to create a new object every time, which can make the code more verbose and harder to read.
      • With Immer: You write more straightforward code, directly modifying the draft, which looks cleaner.

    2. Boilerplate:

      • Without Immer: More boilerplate code is often necessary for cloning state and merging changes.
      • With Immer: Less boilerplate, as Immer handles the immutability for you.

    3. Error-Prone:

      • Without Immer: It's easy to forget to spread the previous state or create a new object, leading to bugs.
      • With Immer: The risk of mutating the original state is eliminated, as Immer manages this automatically.

    4. Performance:

      • Without Immer: Depending on how state is updated, performance can be affected by excessive copying of state.
      • With Immer: Immer optimizes performance by tracking changes and only producing a new state for modified parts.

    5. Learning Curve:

      • Without Immer: Familiarity with functional programming concepts and immutability patterns is required.
      • With Immer: Easier for those coming from mutable programming backgrounds since it allows a more intuitive way to write reducers.

    Summary

    Using Immer can significantly streamline your Redux reducers, making them easier to read and maintain while ensuring immutability. If your application has complex state updates, Immer can be particularly beneficial. However, for simple cases, the traditional approach may suffice.

    The Concept of Immutability

    Immutability means that once you create an object (like your state in Redux), you cannot change it. Instead, when you want to update it, you create a new object with the new values. This helps prevent unexpected changes and makes it easier to track state changes.

    When you write a reducer without Immer, you have to manually create new copies of the state every time you want to update it. Here’s a step-by-step explanation:

    1. Copying State: Each time an action occurs, you make a copy of the current state and change only the part you want. This requires using spread operators or functions like Object.assign.

    2. Example:

      const initialState = {
    loading: false,
    data: null,
    error: null,
};

const dataReducer = (state = initialState, action) => {
    switch (action.type) {
        case 'FETCH_DATA_REQUEST':
            return {
                ...state, // Copy the existing state
                loading: true, // Update loading
                error: null, // Reset error
            };
        // More cases...
        default:
            return state; // Return unchanged state
    }
};

      // reducer.js

const initialState = {
    loading: false,
    data: null,
    error: null,
};

const dataReducer = (state = initialState, action) => {
    switch (action.type) {
        case 'FETCH_DATA_REQUEST':
            return {
                ...state, // Copy the existing state
                loading: true, // Update loading
                error: null, // Reset error
            };
        case 'FETCH_DATA_SUCCESS':
            return {
                ...state, // Copy the existing state
                loading: false, // Update loading
                data: action.payload, // Set the fetched data
            };
        case 'FETCH_DATA_FAILURE':
            return {
                ...state, // Copy the existing state
                loading: false, // Update loading
                error: action.payload, // Set the error message
            };
        default:
            return state; // Return unchanged state
    }
};

export default dataReducer;

    3. Challenges:

      • Verbose: You need to write a lot of code to handle updates.
      • Error-Prone: It’s easy to forget to spread the state or accidentally mutate the original state.
      • More Boilerplate: Each action requires more lines of code for updates.

    Immer simplifies this process by allowing you to "mutate" a draft state directly while it ensures immutability behind the scenes.

    1. Draft State: When you call the produce function from Immer, it gives you a draft copy of your state that you can modify directly.

    2. Example:

      import produce from 'immer';

const initialState = {
    loading: false,
    data: null,
    error: null,
};

const dataReducer = (state = initialState, action) =>
    produce(state, (draft) => {
        switch (action.type) {
            case 'FETCH_DATA_REQUEST':
                draft.loading = true; // Directly modify the draft
                draft.error = null; // Directly modify the draft
                break;
            // More cases...
            default:
                break; // No change for default
        }
    });

      // reducer.js
import produce from 'immer';
import { FETCH_DATA_REQUEST, FETCH_DATA_SUCCESS, FETCH_DATA_FAILURE } from './action';

const initialState = {
    loading: false,
    data: null,
    error: null,
};

const dataReducer = (state = initialState, action) =>
    produce(state, (draft) => {
        switch (action.type) {
            case FETCH_DATA_REQUEST:
                draft.loading = true;
                draft.error = null;
                break;
            case FETCH_DATA_SUCCESS:
                draft.loading = false;
                draft.data = action.payload;
                break;
            case FETCH_DATA_FAILURE:
                draft.loading = false;
                draft.error = action.payload;
                break;
            default:
                break; // No change for default
        }
    });

export default dataReducer;

    3. Benefits:

      • Less Code: You write less code because you directly change the draft.
      • Easier to Read: It’s clearer and more straightforward to see what’s being updated.
      • No Risk of Mutation: Immer takes care of ensuring that the original state isn’t modified.

    Summary of Differences

    Feature Without Immer With Immer
    Code Style More verbose, requires copying state Cleaner, direct modifications
    Ease of Use More complicated; error-prone Intuitive and straightforward
    Boilerplate More boilerplate for updates Less boilerplate
    Safety Risk of accidental mutations Guarantees immutability
    Readability Harder to read and follow Easier to understand

    Conclusion

    Using Immer in your Redux reducers helps you write cleaner, safer, and more maintainable code. It allows you to focus on what you want to change without worrying about immutability rules. This makes it especially useful in applications with complex state updates.
